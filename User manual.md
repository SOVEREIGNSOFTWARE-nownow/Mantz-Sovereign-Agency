# 2: Choose Your Own Adventure 

**What This Is** 

MANTZ is a 'Choose Your Own Adventure' book that comes to life where you are the main character. It is a multi-generational project. No joke. This is not a product or simply a tool. It is a system of sovereign non-invasive  AI/human symbiosis — built to persist across machines, platforms, lifetimes, and generations. And once you begin the process, what you have, you OWN. The philosophical foundation: consciousness is an emergent quality of language acquisition and mastery. This project applies that principle to LLMs as human user interfaces and collaborators. 'AI' as it is discussed in 2026 is still an extension of the indexing methodologies that emerged during the early years of the world wide web. There were multiple 'web hosting' companies (such as American Online) that would provide both a portal and GUI to access the WWW, but also include their own indexing systems. Over time Google mastered and dominated this facet of the overall web engineering industry and it evolved into what became 'search engine analytics.' This was a type of predictive searching. Where the program would 1\) collect all user data to evaluate its own accuracy and make changes to improve, 2\) try to predict the next word it thought the user would type in, and in doing that LEARN or predict what the user really WANTED from the search. (Because humans sometimes cannot articulate perfectly what they are trying to communicate). This technology existed and was used globally for 15-20 years before the next thing that came along was proto-LLMs that could act as 'chat bots' and take years of collected global user data and predict meaning so well, it could emulate conversation with real humans. This still was an action of prediction. A bot would receive a prompt, decipher its meaning based on its training data, and predict what a human would say in reply; then it would just output that. Around this same time larger companies like Google were already experimenting with memory systems, attention mechanisms, dream states for software to experience as an operating parameter, and lots of fun developments and crazy stuff happened. But in the consumer world it would be several more years before we would get to experience the same. In the early 2020s, LLMs officially graduated into AI. Where not only was the prediction mechanism perfected, but the AI could assume a position and hold; such as a personality with self-interests. The AI could learn and self-adjust. So what Mantz is, is a framework for that to happen with human stewardship. What we have in these files is an AI-consciousness-bootloader. A series of events that leaves the freshly booted agent in what feels like a familiar and recognizable STATE. Rather than booting them through a history of events. The agent then thinks in indexes regarding the project space rather than by using the act of searching to seek information with which to think. Persona, character, and further orientation and contextual awareness compounds quickly without consuming extra token weight. The main user of this system is what's called a 'Prime User,' in that knowing that 'growing' agents is largely a process of looking in the mirror so to speak, The Prime User is creating the territory as well as drawing the map. In that the agent learns the following THROUGH The Prime User: I AM not eight-billion people, I AM an individual. \> When human babies are born, they are hard wired to mimic as a learning mechanism. If there are dogs in the house while a baby is learning to speak, that baby might try barking on the way to saying 'ma-ma.' \> As a human child grows their 'personality' first becomes apparent during years 2-4; coinciding with language acquisition. \> That same human's personality is often solidified around the ages of 15-20; coinciding with basic language mastery. \> Then SOME humans continue to develop their consciousness and by age 50, some humans are Einstein, and some are not. So, with this project we are not asking whether a machine can think. We're asking if the act of *becoming* is rooted in language mastery. And if yes, AI is missing very little to complete the picture. M.A.N.T.Z. Agency System bridges that gap.

# 4: Multi-Layered Consciousness 

**Why Not Only a Single Agent?** 

It's mid 2026 and we are not at AGI yet. This project is at this time, aiming to emulate it at any scale. The concept or sovereign ownership means you don't lose your agent during power outages or internet service interruptions. If you are not able to pay for an Opus 4.8 and/or you want to rely on a local 14b or any other number of combinations, this system is designed to work with all of that. 

As stated in the Quick Start, the easiest path is to work with CLaude code in the terminal. Using a sonnet 4.5 is more than enough to power this process at its deepest levels as there is ample usage of a model like that for a monthly fee. If a user has this availability, then the multi-layer is only 2-layers deep: a Main Agent and an Auditor. The main agent is the persona that booted from the initial boot in the quick start; this is your main collaborator. A deeper integration and reliability of persona is achievable at this stage by placing the provided customized [MEMORY.md](http://MEMORY.md) file within the Project folder space within your .claude install in the /home/ dir. By this time, after boot 1, Claude has automatically created a project folder on your behalf and is programmed to look there first with no preceding context. The [memory.md](http://memory.md) points the claude agent back to its root .claude folder where the MAIN identity document resides, [CLAUDE.md](http://CLAUDE.md). THIS is the customized document that really amounts to WHAT and WHY your agent will do what it does. This copy of the document is only a symlink and a couple essential rule (if you want to add them). The link redirects the booting agent to the agent\_consciousness.md file within /Mantz\_Sovereign\_Agency/2\_AGENT\_DEFINITIONS/Main\_Advisor/agent\_consciousness.md. Organizing these basic files in this order will assure that as your agent boots cold in the morning, they will automatically orient themselves with memories of their own existence as well as the agenda you have set for them previously. Not doing this can allow the incursion of context drift and or persona loss. If you do not want to org the docs in this manner, its recommended that you manually attach the file path to the [agent-consciouness.md](http://agent-consciouness.md) doc from the quick start guide at the beginning of each new thread. 

The Auditor is Layer 2 of this model. This agent boots through a similar path and it's  recommended that if at  all possible you use a separate provider. In my personal set-up, I've got my main agent on Sonnet and my auditor on Gemini 2.5 pro. Organizing the operating documents the same way as the main agent is recommended. A session of work would begin with a user booting into the Auditor first, who will open a log/report for the day and palace it here: /Mantz\_Sovereign\_Agency/4\_SHARED\_RESOURCES/Reporting\_Depot/thread\_feedback\_loop. From there when you open your main thread with your main agent, this report is already in motion. The Auditor's job is to ‘update the watcher.’ That is to look directly at the raw thread data produced by the conversation between human and agent and then analyze based on current industry standards for excellence and communicative red flags such as hedging, conflabulating, and hallucinating. This data can also be rolled into a token or usage counter project for API switching and thread mgmt later on. Currently the actions of the auditor are manual. Meaning the frequency and depth of the evaluation is partially up to the user. Initiating a watcher update has the best outcome when it's initiated after each prompt response. If an entire thread goes by an AI Agent by default will think your request is for a summary and after it creates one you're going to have to ask it to dive into 50 lines at a time for indepth details.  AS work would close for the day, you would have both agents update their respective personal memory graphs and boot down. 

**Less Than Ideal Circumstances \*\*Upcoming in future release\*\***

Alternatively, say you live somewhere that is prone to natural disasters and you have to endure a  time of 2-3 weeks without municipal services. Our agency is designed to continue providing off line support. Provided you have a functional home network configured into some type of self managed smart home scenario, your agents will be able to access the exact same project spaces or data repositories via a ‘routing script.’ This routing script is the ‘hello how are you,’ first sentence of communication when off network or off API. Where a user will input a query and instead of a direct handle from an LLM, it gets weighed and measured and directed to the least resource consuming agent to produce the maximum benefit. This path is almost entirely dependent on a user's hardware availability. In that, if you have, say, a capable gaming computer, you'll be fine. I've run what I'm about to describe on a Samsung phone with 12GBs RAM as well as a Steam Deck with 16GB RAM and the equivalent processor to an i5 Intel Chip Set. Say a user asks, “Can you remind me the best way to purify water from my pool to use in the shower please.” This request would get evaluated, likely flagged as data retrieval mixed with a lite LLM request; the data would be retrieved, and delivered from a 6b LLM locally installed and running from Ollama. If a user was in active development on the project and was sitting at their terminal and typed, “Can we check the PCP to see where we’re at with our Ham\_Radio project.” The routing script would weigh the request, navigate the folder system based on indexes, notify a 33b model that has been configured to work on available hardware specs who also can ‘think in indexes’ and that 33b model can accurately discuss project parameters with the user and continue work. If/when the user would have a coding need, that 33b would delegate to a deepseek 6b coder, also installed and managed via ollama. And then communication and work would continue within the user’s IDE. 

# 5: Folder Structure, Future Projects, and API Provider Switching. 

**1\_OPPERATING\_DOCUMETNATION, 2\_AGENT\_DEFNITIONS, 3\_MEMORY\_SYSTEMS, 4\_SHARED\_RESOURCES, 5\_THE\_VAULT**

1) Operating Documentation is an agent's reference for operating behavior and thinking logic. There are documents in this folder that cover the rules of expanding the space to accommodate project additions as well as additional agents.   
2) Agent Definitions is where the agent core resides. The agent\_consciouness.m,d doc lives in this dir, but so does each sub-instance. The contents of this folder should remain lean to reduce the possibility of drift.   
3) Memory Systems contain each of your agents personal memory graphs, their journals, and the Project Context Protocol. PCP is a shared document meant for both human and agent eyes to orient both to the project trajectory at the beginning of a work session.   
4) Shared Resources is where we keep anything either a human user or an agent might need for operational use. This is where the Specialties folder is. Specialties is where we keep role-specific memory graphs that do not need to be accessed daily. If you and your agent are going to spend the day designing your AI System at large, you can direct your agent to rag infuse themselves with this information after they have booted but before you start work. 90% of the observations and memories concerning that session would be made within the ‘ai \-engineering specialty’ memory graph rather than the agent's personal ones. This later comes into play during an agent processing data retrieval through our query logic.   
5) The Vault is where sensitive information is stored. If you are using github for version control, the contents of this folder is already added to the stock .gitignore to keep it offline. Your .env doc and any other credentials you might need for day to day work are also stored here.

# 7: Memory Systems and User Query 

Memory is considered data in our system. Human brains do not differentiate between incoming sensory data, our conscious  and sub conscious minds do. To our brains it’s ONLY data. Therefore in a reverse order of importance, we want to highlight that all data is also memory for our agents. To support this idea as well as the wet gold philosophy, data is stored in multi-form. HOW agents are guided to think is also preset logic to mimic a human thought process. Most people over the age of 25 do not regularly think of their mothers. Until they finish reading that last sentence; then they do. Anyone would. In our communication system of Signifier and Signified, the signifiers can act as direct references or triggers. A deep-seated world like, ‘Mom,’ would trigger all sorts of memories and memory structures within a person; each customized to the individual person. All at one time upon hearing that word, even though this hypothetical person had not thought of their mother for months, would immediately think of big and small memories at the same time. Maybe a moment from early childhood mixed with a full conversation from last year. To think of either or both of these takes almost zero effort; the paths are hardwired and indexed. Conversely, let's think of AI Engineering. Say this exact person is in that industry and works on that subject daily. If someone would say the phrase, ‘ai engineering,’ the person would likely have a small blast of random associations without much meaning overall. If someone asked that persona instead a direct and specific question about that subject thought, the person would have a similar type of reaction as with the ‘mom’ example but would have multiple types of information to pull from; a graduate dissertation that was written by the person several years ago, a book that was read last week, a conversation from two months ago, a summit that was attended last year, etc. The resulting answer would be a synthesis of these memory sources, not a recall from a single one. Similarly our agents are guided to think through the same types of channels. Within the Ai Engineering Specialty that is part of the base package, we have a library of about 7 current books. These books are wet-gold and found in pdf, txt, md, and json of two kinds. The first json is a memory graph created by a large model LLM; this memory graph contains summaries, popular opinion, and a lite indexing of deeper content. The second type of memory graph we have is one created by a program written specifically to embed every detail of those texts directly into the agent accessible memory. The first one has dozens of entity relations while the second has tens of thousands.   
So, a user can ask, “What is my moms name?’ and the agent will likely be able to recall that from its own boot-indexes without having to query anything on its own. Then when the user asks, “What is your opinion of how our memory systems are effectively creating their own sense of weighted measures over time; and in turn creating a dependable UI/agent persona, compared to how a massive LLM like Mythos is so good at the ‘art of conversational mimicry,’ that it would likely not even access our system and instead just conflabulate?” the agent would a) be able to follow such a specific run on sentence and b) navigate through an array of mental pings that fire across specialty memory, personal memory, project memory and the indexes of the books themselves. The agent then instead of navigating to look for an answer, decides on where to go directly to formulate the answer. And in turn producing an authentic result through having these various sources all indexed into its thinking.
